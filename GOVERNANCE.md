# Governance

Aegora is community-led. The goal is to avoid a single-maintainer bottleneck and create a clear path from contributor to maintainer.

## Roles

### Contributor

Anyone who participates through issues, discussions, documentation, examples, tests, code, review, or community support.

### Reviewer

A trusted contributor who helps review pull requests, triage issues, mentor contributors, and guide technical discussions.

### Maintainer

A reviewer with merge rights and release responsibilities.

Maintainers are expected to act as stewards of the project, not owners of the community.

## Contributor Path

A contributor may be invited to become a reviewer after sustained, constructive participation such as:

- Meaningful merged pull requests
- Helpful documentation or examples
- Consistent issue triage
- Thoughtful technical discussion
- Mentorship and community support

A reviewer may be invited to become a maintainer after showing:

- Sound technical judgment
- Respectful communication
- Consistent participation
- Care for project stability
- Alignment with the community model

## Decision Making

Small changes can be accepted with one maintainer approval.

Larger changes should use an RFC-style issue or discussion before implementation. Examples include:

- Public API changes
- Breaking changes
- New package architecture
- Runtime behavior changes
- Governance or release policy changes

Large changes should usually have:

- A clear proposal
- At least seven days for community feedback when practical
- At least two maintainer approvals once the maintainer group has enough members

## Maintainer Responsibilities

Maintainers should:

- Review contributions respectfully
- Explain declined changes clearly
- Keep the roadmap visible
- Protect project stability
- Encourage new contributors
- Avoid using authority to shut down good-faith discussion

## Conflict Resolution

When disagreement happens, the project should prefer:

1. Clarifying the actual technical or community concern
2. Identifying tradeoffs
3. Looking for reversible decisions
4. Documenting the decision
5. Escalating to maintainers only when needed

Personal attacks, harassment, and bad-faith participation are not acceptable.

## Project Direction

Aegora's priorities are:

- Reliable TypeScript APIs
- Clear contracts and documentation
- Production readiness
- Friendly contribution paths
- Transparent governance
- Long-term maintainability
