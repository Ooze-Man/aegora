# Mentorship

Aegora is designed to be a place where contributors can learn while building real framework features.

## What Mentorship Means

Mentorship does not mean maintainers write the code for contributors.

It means contributors can expect enough context to make progress:

- What problem are we solving?
- Why does it matter?
- Which files are relevant?
- What behavior should change?
- How can the work be tested?
- What would a strong pull request include?

## Mentored Issues

Issues labeled `mentored` should include:

```md
## Goal

## Background

## Relevant files

## Acceptance criteria

## Suggested test coverage

## Optional stretch goal
```

A contributor can ask for mentorship by commenting:

```txt
I'd like mentorship on this.
```

A maintainer, reviewer, or experienced contributor should respond with next steps when available.

## Good First Issues

Good first issues should be small, clear, and scoped. They should not require deep knowledge of Aegora internals.

Good examples:

- Improve an error message
- Add missing docs for an existing option
- Add a reproduction test for a known bug
- Add an example app for an existing pattern
- Improve type tests around an edge case

Poor examples:

- Redesign routing
- Replace the type system
- Rewrite the build pipeline
- Add a major new plugin without design discussion

## Review Expectations

Mentored issues should receive review that is practical and kind.

Useful review comments explain:

- What should change
- Why it matters
- Where to look
- Whether the suggestion is required or optional

## Becoming a Mentor

You do not need to be a maintainer to mentor.

Experienced contributors can help by:

- Answering setup questions
- Pointing to relevant files
- Explaining test failures
- Reviewing docs changes
- Helping turn vague issues into actionable tasks

The goal is to create more contributors, not gatekeep knowledge.
